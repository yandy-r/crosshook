# Pattern Research: remove-native

## Architectural Patterns

- **Thin packaging scripts over shared path helpers**: Build scripts set `ROOT_DIR`, source `scripts/lib/build-paths.sh`, parse flags with `case`, and use small local helpers like `die()`/`log()`. Keep this shape when splitting the Flatpak binary build from AppImage bundling. Examples: `scripts/build-flatpak.sh`, `scripts/build-native.sh`, `scripts/lib/build-paths.sh`.
- **Flatpak packaging stages committed assets into a temporary build root**: `scripts/build-flatpak.sh` stages `crosshook-native`, runtime helpers, icons, desktop file, metainfo, and the manifest into a `mktemp` directory, then runs `flatpak-builder` and `flatpak build-bundle`. Follow this staging pattern if the script stops calling `build-native.sh --binary-only`.
- **Release workflow currently uses a native/AppImage job as Flatpak input producer**: `.github/workflows/release.yml` job `build-native` builds the AppImage, exports `dist/crosshook-native`, uploads AppImage/CLI assets, and separately uploads `flatpak-staging-assets`; `build-flatpak` consumes that staging artifact. A Flatpak-only release should keep the explicit artifact handoff but rename/re-scope the producer around the release binary or Flatpak staging, not AppImage.
- **Tauri config owns desktop shell build, not distribution policy**: `src/crosshook-native/src-tauri/tauri.conf.json` has normal Tauri build settings plus `bundle.targets: ["appimage"]`, `bundle.resources`, and `bundle.icon`. Removing AppImage should avoid breaking `frontendDist` embedding, which `scripts/build-native.sh --binary-only` currently preserves via `tauri build --no-bundle`.
- **Runtime Flatpak behavior is isolated in core modules**: Flatpak detection, host command wrapping, XDG override, and host filesystem helpers are centralized under `src/crosshook-native/crates/crosshook-core/src/platform/`. Startup migration mode is selected in `src/crosshook-native/src-tauri/src/lib.rs`, and first-run import lives in `src/crosshook-native/crates/crosshook-core/src/flatpak_migration/`.
- **Docs are layered by purpose**: User-facing setup lives in `README.md` and `CONTRIBUTING.md`; distribution implementation notes live in `packaging/flatpak/README.md`; durable architecture decisions live in `docs/architecture/adr-*.md`; implementation plans/reports live under `docs/prps/` or `docs/plans/`.

## Code Conventions

- **Shell scripts**: Use `#!/usr/bin/env bash`, `set -euo pipefail`, uppercase configuration vars, uppercase mode flags (`REBUILD=0`), arrays for command args, and `[[ ... ]]` / arithmetic `(( ... ))`. Unknown arguments fail through `die` or usage output. Examples: `scripts/build-flatpak.sh`, `scripts/prepare-release.sh`, `scripts/lint.sh`.
- **Environment knobs are documented in usage blocks**: Build scripts list both flags and env vars in heredoc usage text. Any replacement for `build-native.sh --binary-only` should document `DIST_DIR`, `CARGO_TARGET_DIR`, `CROSSHOOK_BUILD_EPHEMERAL`, and any Flatpak-specific env vars in the same style.
- **Root package scripts wrap repo scripts**: Root `package.json` exposes `build:binary`, `build:appimage`, `flatpak:build`, `flatpak:install`, and release/lint wrappers. Removing AppImage should update script names so package scripts map cleanly to the surviving Flatpak path.
- **Native package scripts stay frontend/Tauri scoped**: `src/crosshook-native/package.json` owns Vite, Tauri CLI, Vitest, Playwright smoke, Biome, and typecheck scripts. Distribution changes should avoid inventing frontend package scripts unless the command truly belongs under the Tauri app package.
- **CI names are descriptive and split by concern**: `.github/workflows/lint.yml` separates `rust`, `typescript`, `shell`, and `smoke` jobs. `.github/workflows/release.yml` uses job names matching artifact responsibilities. Rename `build-native` if it no longer builds a native distribution artifact.
- **Tauri IPC/Core boundaries are preserved**: Distribution cleanup should not move business logic into `src-tauri`; existing rules keep business logic in `crosshook-core` and Tauri as thin IPC/startup glue.
- **Existing path names may lag architecture**: `scripts/check-host-gateway.sh` still names `platform.rs` in comments and `PLATFORM_RS`, but the current implementation is a `platform/` module tree. When updating AppImage-era text, verify current paths before preserving old wording.

## Error Handling

- **Scripts fail fast but aggregate linter failures intentionally**: Build/release scripts use `set -euo pipefail` plus `die`. `scripts/lint.sh` collects failures into `EXIT_CODE` so all selected checks can run before exiting.
- **Preflight checks are explicit**: `scripts/build-flatpak.sh` checks for `flatpak-builder`, `flatpak`, manifest/desktop/metainfo files, and release binary presence before staging. Preserve these checks when replacing the binary producer.
- **Mutually exclusive modes are guarded early**: `scripts/build-flatpak.sh` rejects `--rebuild` with `--skip-build`. Keep this pattern for any new `--rebuild`/`--skip-build` semantics after AppImage deletion.
- **Optional validators warn by default and fail in strict mode**: `scripts/build-flatpak.sh` runs `desktop-file-validate` and `appstreamcli validate` when available; `--strict` or `CROSSHOOK_FLATPAK_VALIDATE_STRICT=1` turns those warnings into failures.
- **Release prep requires a clean, version-aligned state**: `scripts/prepare-release.sh` fails if the tree is dirty, the tag exists, `git-cliff` is missing, manifests are missing, or generated release notes fail validation. Keep the version sync and changelog validation path even if distribution assets change.
- **Runtime startup logs and continues for migration failures**: `src/crosshook-native/src-tauri/src/lib.rs` logs/eprints Flatpak migration failures but continues startup; `flatpak_migration::run()` collects subtree copy errors without aborting the whole import. Do not turn distribution cleanup into a data-loss migration rewrite unless explicitly planned.

## Testing Approach

- **Core Rust validation**: For runtime/platform changes, run `cargo test --manifest-path src/crosshook-native/Cargo.toml -p crosshook-core`. Relevant test coverage includes `src/crosshook-native/crates/crosshook-core/tests/flatpak_migration_integration.rs`, `src/crosshook-native/crates/crosshook-core/src/platform/tests/*`, and launch/runtime helper tests.
- **Shell and policy guards**: Run `./scripts/lint.sh --shell --host-gateway` or full `./scripts/lint.sh` after script/workflow edits. The host-gateway guard is still important because Flatpak remains the only distribution path and host-tool routing matters more, not less.
- **Flatpak packaging validation**: Use `./scripts/build-flatpak.sh --strict` for local packaging checks where Flatpak tooling is available. Metadata-only checks are `desktop-file-validate packaging/flatpak/dev.crosshook.CrossHook.desktop` and `appstreamcli validate packaging/flatpak/dev.crosshook.CrossHook.metainfo.xml`.
- **Release workflow validation target**: After removing AppImage assets, the release workflow should still prove version alignment, `crosshook-core` tests, production bundle mock sentinel, CLI packaging if CLI remains shipped, Flatpak staging, Flatpak metadata validation, bundle build, and release upload with `fail_on_unmatched_files: true`.
- **Frontend validation remains separate**: Distribution cleanup rarely needs UI tests, but the standard frontend gates are `npm run typecheck`, `npm test`, and `npm run test:smoke` from `src/crosshook-native/`; `docs/TESTING.md` is the canonical testing guide.
- **Production mock sentinel must survive AppImage removal**: The current sentinel in `.github/workflows/release.yml` scans `src/crosshook-native/dist/assets/*.js` after the Tauri build. It is described as AppImage-related in docs, but the security value applies equally to Flatpak.

## Patterns to Follow

- **Rename around responsibility, not nostalgia**: Prefer names like `build-release-binary`, `stage-flatpak-assets`, or `build-flatpak` over broad deletion of every `native` reference. In this repo, "native" can mean the Linux/Tauri app or AppImage packaging depending on context.
- **Keep `tauri build --no-bundle` semantics for Flatpak inputs**: Flatpak currently depends on a production-embedded release binary. A plain `cargo build --release` is explicitly called out as insufficient in `scripts/build-native.sh`.
- **Remove AppImage-only surfaces together**: Candidate surfaces include `scripts/build-native-container.sh`, `scripts/build-native-container.Dockerfile`, AppImage branch/functions in `scripts/build-native.sh`, `scripts/generate-crosshook-desktop.sh`, `scripts/lib/sync-tauri-icons.sh` if it only serves Tauri AppImage icons, `crosshook_appimage_bundle_dirs()` in `scripts/lib/build-paths.sh`, `bundle.targets: ["appimage"]` in `src/crosshook-native/src-tauri/tauri.conf.json`, root package scripts `build:appimage`, release workflow AppImage upload paths, and AppImage user docs.
- **Preserve asset generation needed by Flatpak**: `scripts/generate-assets.sh` is not AppImage-only; Flatpak uses `assets/icon-128.png`, `assets/icon-256.png`, and `assets/icon-512.png`.
- **Update docs that encode AppImage as primary**: High-priority docs include `README.md`, `CONTRIBUTING.md`, `CLAUDE.md`, `AGENTS.md`, `packaging/flatpak/README.md`, `docs/internal-docs/local-build-publish.md`, `docs/architecture/adr-0001-platform-host-gateway.md`, `docs/architecture/adr-0003-proton-download-manager.md`, and `docs/architecture/adr-0004-flatpak-per-app-isolation.md`.
- **Be careful with first-run migration wording**: `flatpak_migration` imports "host AppImage" data today. If AppImage is no longer distributed, implementors should decide whether this remains a legacy import path for existing users or becomes generic "legacy host XDG data" wording. Do not remove it without a migration/backward-compatibility decision.
- **Keep Flatpak runtime version in lockstep**: If touching Flatpak packaging, keep `runtime-version` in `packaging/flatpak/dev.crosshook.CrossHook.yml`, `RUNTIME_VERSION` default in `scripts/build-flatpak.sh`, and the release workflow container image tag aligned.
- **Use existing validation instead of new bespoke checks**: Prefer extending `scripts/lint.sh`, `scripts/build-flatpak.sh --strict`, release workflow artifact checks, and existing Rust tests over adding one-off validation scripts.
