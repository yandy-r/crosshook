# Recommendations: launcher-delete

## Executive Summary

The current launcher export system is fire-and-forget: it writes a shell script and `.desktop` entry to well-known paths but never records what was exported or which profile it belongs to. This means profile deletion and renaming have no way to cascade to launcher artifacts today. The recommended approach is to introduce a lightweight launcher manifest (a TOML registry at `~/.local/share/crosshook/launchers/manifest.toml`) that tracks every exported launcher's paths, owning profile name, and slug. All lifecycle operations -- delete, rename, orphan detection, and integrity checks -- become straightforward lookups against this manifest rather than fragile filesystem scans.

The primary risk is accidental deletion of user-modified launcher files. Mitigation is to always verify file content (checking for the `# Generated by CrossHook` watermark) before removing or overwriting, and to surface confirmation prompts in the UI for destructive operations.

## Implementation Recommendations

### Recommended Approach

Introduce a `LauncherManifest` model in `crosshook-core/src/export/` that persists a list of `LauncherRecord` entries. Each record captures the profile name, launcher slug, script path, desktop entry path, and an export timestamp. The manifest is written atomically alongside every export and consulted on every delete/rename.

Profile lifecycle operations (`profile_delete`, and a new `profile_rename`) should call into a new `launcher_lifecycle` module that reads the manifest, identifies associated launchers, performs the filesystem operations, and updates the manifest. The Tauri command layer orchestrates: `profile_delete` calls the existing store delete, then calls launcher cleanup; `profile_rename` calls store save-under-new-name + delete-old, then calls launcher rename.

On the frontend, the `LauncherExport` component gains a management section showing the current launcher status (exported, stale, missing) with Delete and Rename buttons that invoke new IPC commands directly, independent of the profile lifecycle flow.

### Technology Choices

| Component         | Recommendation                                                           | Rationale                                                                                                           |
| ----------------- | ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------- |
| Launcher tracking | TOML manifest file at `~/.local/share/crosshook/launchers/manifest.toml` | Consistent with existing TOML persistence for profiles and settings; human-readable; atomic write via temp + rename |

| Orphan detection  | Manifest scan + filesystem cross-reference on app startup                | Catches launchers whose profiles were deleted outside the app or renamed manually                                   |
| Concurrency       | No special handling needed in v1                                         | Single-user desktop app; Tauri command handlers are already serialized per-state                                    |

### Phasing Strategy

1. **Phase 1 - Foundation (Manifest + Core Delete)**
   - Introduce `LauncherRecord` and `LauncherManifest` models in `crosshook-core/src/export/`
   - Add `LauncherManifestStore` with `load`, `save`, `add_record`, `remove_record`, `find_by_profile` methods
   - Add `delete_launcher` function that removes script + desktop entry from disk and updates manifest
   - Retrofit `export_launchers` to write a manifest entry on every successful export
   - Wire `profile_delete` Tauri command to cascade launcher deletion via the manifest

2. **Phase 2 - Rename + Manual Management**
   - Add `ProfileStore::rename` method
   - Add `rename_launcher` function that regenerates script content with updated display name, moves files to new slug paths, and updates manifest
   - Wire `profile_rename` through Tauri IPC and add a rename action to the frontend profile editor
   - Add `delete_launcher` and `rename_launcher` IPC commands for direct manual management from `LauncherExport`
   - Add launcher status indicator to the LauncherExport panel (shows "Exported" / "Not exported" / "Stale")

3. **Phase 3 - Integrity + Orphan Management**
   - Add startup integrity check that cross-references manifest entries with filesystem
   - Add orphan detection (manifest entries whose script/desktop files are missing, or disk files with no manifest entry)
   - Add a launcher management section in Settings panel listing all known launchers with bulk delete
   - Add "Stale launcher" warning when profile fields that affect launcher content have changed since last export

### Quick Wins

- **Cascade delete on profile_delete**: Even without a manifest, the current slug derivation is deterministic. A quick win is to compute the expected launcher paths from the profile's launcher display name (or game name) and attempt to delete them during `profile_delete`. This provides immediate value while the manifest system is being built.
- **Launcher status indicator**: The `SteamExternalLauncherExportResult` returned by `export_launchers` contains the exact paths. Persisting just these two paths (script + desktop entry) in the profile TOML as a `[launcher_export]` section gives instant status checking with minimal new infrastructure.
- **Confirmation dialog**: Add a "This will also delete the exported launcher" confirmation when deleting a profile that has a known launcher. Low effort, high safety value.

## Improvement Ideas

### Related Features

- **Launcher status tracking**: Add a `launcher_export` section to `GameProfile` TOML that records `script_path`, `desktop_entry_path`, `exported_at`, and `launcher_slug`. This gives per-profile launcher awareness without a separate manifest file (trade-off: less centralized than a manifest, but simpler for v1).
- **Auto-detection of orphaned launchers**: On app startup, scan `~/.local/share/crosshook/launchers/` and `~/.local/share/applications/crosshook-*.desktop` for files that do not match any known profile. Surface these in Settings with a "Clean up" action.
- **Launcher integrity checking**: Compare the launcher script's embedded paths (trainer path, proton path, prefix path) against the profile's current values. If they differ, mark the launcher as "stale" and prompt re-export.
- **Bulk launcher management**: A launcher management panel in Settings that lists all exported launchers with checkboxes for bulk delete, similar to how the community browser lists available profiles.
- **Launcher sync status in profile list**: Add a small icon or badge next to each profile name in the profile selector dropdown indicating whether a launcher is exported, missing, or stale.

### Future Enhancements

- **Auto-re-export on profile save**: When saving a profile that has an exported launcher, offer to re-export automatically if the launcher-relevant fields changed. Complexity: medium, value: high for power users.
- **Launcher export history**: Track previous exports so users can see what changed. Complexity: medium, value: low initially.
- **Desktop environment integration**: After exporting, optionally run `update-desktop-database` to refresh the application menu. Complexity: low, value: medium for non-Steam-Deck users.
- **Profile rename propagation to community taps**: When renaming a profile that was shared via a community tap, consider whether the tap reference needs updating. Complexity: high, value: low initially.

## Risk Assessment

### Technical Risks

| Risk                                                                                  | Likelihood | Impact | Mitigation                                                                                                                                                                                              |
| ------------------------------------------------------------------------------------- | ---------- | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Deleting wrong launcher file (user manually created a `.desktop` with similar naming) | Low        | High   | Verify `# Generated by CrossHook` comment/watermark in both script and desktop entry before deletion; never delete files that fail this check                                                           |
| Race condition between profile delete and concurrent export                           | Very Low   | Medium | Single-user desktop app with serialized Tauri command handlers; no special locking needed in v1                                                                                                         |
| Manifest becomes stale or corrupted                                                   | Low        | Medium | Treat manifest as advisory; always verify filesystem state before acting; rebuild manifest from filesystem scan as a recovery path                                                                      |
| Slug collision (two different profile names produce the same launcher slug)           | Low        | Medium | `sanitize_launcher_slug` is lossy; manifest records the exact profile name, so deletion targets the correct record even if slugs collide; warn on export if slug already exists for a different profile |
| Filesystem permissions prevent deletion (read-only mount, restricted home)            | Low        | Medium | Surface clear error messages including the exact path and permission issue; do not silently swallow failures                                                                                            |
| Profile renamed outside CrossHook (manual TOML file rename)                           | Medium     | Low    | Orphan detection on startup catches this; the old manifest entry becomes orphaned and the new profile has no launcher record                                                                            |
| AppImage/Flatpak sandbox restricts file operations in `~/.local/share/applications/`  | Low        | High   | Currently works (export already writes there); document the requirement; test in sandboxed environments                                                                                                 |

### Integration Challenges

- **`profile_delete` command needs launcher awareness without tight coupling**: The profile delete Tauri command currently delegates to `ProfileStore::delete` which knows nothing about launchers. The cleanest approach is to handle launcher cleanup at the Tauri command level (after the store delete succeeds), not inside the store itself. This keeps `crosshook-core` modules loosely coupled.
- **Frontend state synchronization**: When a launcher is deleted (either via profile delete cascade or manual action), the `LauncherExport` component needs to know. Either re-query launcher status after any mutation, or use Tauri events to push state changes.
- **Rename requires re-generating file content**: A launcher rename is not just a file move -- the script content embeds the display name in comments, and the desktop entry contains `Name=`. Both files must be regenerated, not just renamed on disk.

### Security Considerations

- **Path traversal in launcher slug**: The `sanitize_launcher_slug` function already strips path separators, but any new code that accepts user-provided paths for deletion must validate that the target is inside `~/.local/share/crosshook/launchers/` or `~/.local/share/applications/` and matches the `crosshook-` prefix pattern.
- **Symlink attacks**: Before deleting a launcher file, verify it is a regular file (not a symlink) to prevent symlink-following attacks that could delete unrelated files.
- **Manifest tampering**: If an attacker modifies the manifest to point at arbitrary paths, the delete operation could remove important files. Mitigation: always validate that target paths are within expected directories and carry the CrossHook watermark.

## Alternative Approaches

### Option A: Manifest Registry Approach

A dedicated `manifest.toml` file in `~/.local/share/crosshook/launchers/` tracks every exported launcher as a structured record (profile name, slug, paths, timestamp).

- **Pros**: Single source of truth for all launchers; enables efficient lookup by profile name; supports orphan detection; decoupled from profile TOML schema; scales to bulk operations
- **Cons**: New file to manage; can become stale if the app crashes between file write and manifest update; adds complexity to the export path

### Option B: Profile-Embedded Tracking

Add a `[launcher_export]` section directly to each profile's TOML file with `script_path`, `desktop_entry_path`, and `exported_at` fields.

- **Pros**: No new files; launcher state is always co-located with the profile that owns it; trivially survives profile rename (the section moves with the profile); no orphan problem for profile-owned launchers
- **Cons**: Cannot detect launchers for deleted profiles (the TOML is gone); requires scanning all profiles to list all launchers; does not support launchers exported outside the profile flow; orphan detection requires filesystem scanning anyway

### Option C: Filesystem Scanning Approach

No persistent tracking. On every operation, scan `~/.local/share/crosshook/launchers/` for `*-trainer.sh` files and `~/.local/share/applications/crosshook-*-trainer.desktop` files. Derive the owning profile by reversing the slug.

- **Pros**: Zero new state; always accurate with the filesystem; no stale manifest problem
- **Cons**: Slug-to-profile mapping is lossy and irreversible (multiple profile names can produce the same slug); slow for large numbers of launchers; cannot distinguish CrossHook-generated files from similarly named user files without content inspection; fragile if naming conventions change

### Option D: Hybrid (Profile-Embedded + Lightweight Manifest)

Store launcher export metadata in the profile TOML (Option B) for the primary lookup path, and maintain a lightweight manifest (Option A) as a secondary index that is rebuilt from profile data on startup.

- **Pros**: Best of both worlds -- fast lookup, resilient to manifest corruption, profile rename is trivial, orphan detection via manifest rebuild
- **Cons**: Most complex to implement; two sources of truth require reconciliation logic

### Recommendation

**Option A (Manifest Registry)** for v1, with a future migration path toward Option D if profile-embedded tracking proves valuable. Rationale:

- The manifest is the only approach that can track launchers for profiles that have been deleted (needed for the cascade-delete use case -- the manifest tells us what to clean up).
- Profile-embedded tracking (Option B) is a natural Phase 2 addition that enhances per-profile status indicators without replacing the manifest.
- Filesystem scanning (Option C) should be reserved as a recovery/rebuild mechanism, not the primary lookup path.

## Key Decisions Needed

- **Manifest vs. profile-embedded tracking**: Should launcher state live in a separate manifest file, or be embedded in each profile TOML? This report recommends manifest-first (Option A), but the hybrid (Option D) has merit if profile-level status indicators are a priority.
- **Automatic vs. prompted cascade delete**: When deleting a profile with an exported launcher, should the launcher be deleted automatically, or should the user be prompted with a confirmation dialog? Recommendation: prompt in the UI, but provide an `auto_delete_launchers` setting for power users.
- **Slug collision handling**: When two profiles produce the same launcher slug, should the second export overwrite the first, error out, or append a disambiguator? Recommendation: warn and require confirmation to overwrite; update the manifest to point at the new owner.
- **Rename scope**: Should profile rename only update the TOML filename, or also update the `game.name` and `steam.launcher.display_name` fields inside the profile? Recommendation: rename changes the profile file name and launcher artifacts only; internal profile fields remain unchanged unless the user explicitly edits them.

## Open Questions

- Should launcher deletion be reversible (move to a trash/archive directory instead of permanent delete)?
- Should the manifest track launchers exported by the CLI (`crosshook-cli`) in addition to the GUI, and if so, should the CLI also be able to read/write the manifest?
- Is there a need to support multiple launchers per profile (e.g., different launcher names or configurations for the same game)?
- Should the startup integrity check run silently, or should it surface a notification/toast when issues are found?
- Are there plans for macOS support that would require `.app` bundle or `launchd` integration instead of `.desktop` entries, and should the manifest schema account for platform-specific launcher types from the start?
