## Summary

The worktree is in generally good shape and all validation I ran passed, but the new umu guidance flow has two functional regressions. The persisted dismissal timestamp is never consulted when readiness is recomputed, and the browser dev-mode save path does not actually preserve the new optional field semantics this change depends on.

## Findings

### CRITICAL

### HIGH

- **[F001]** `src/crosshook-native/src-tauri/src/commands/onboarding.rs:10` — `dismiss_umu_install_nag` persists `install_nag_dismissed_at`, but `check_readiness` still calls `check_system_readiness()` without reading settings, so the dismissal is never honored. After a user dismisses the reminder, the next `Run Checks` or reopened wizard will recreate `umu_install_guidance`, which makes the new stored timestamp dead data.
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Thread `SettingsStore` into `check_readiness` (or add a settings-aware core readiness variant) and suppress `umu_install_guidance` when `install_nag_dismissed_at` is set; add a regression test for dismiss -> rerun -> no guidance.

- **[F002]** `src/crosshook-native/src/types/settings.ts:74` — `toSettingsSaveRequest()` relies on `undefined` meaning “field omitted”, but browser dev mode forwards the raw args object into the mock layer, where the payload is shallow-merged. That means `install_nag_dismissed_at: undefined` no longer behaves like the Rust backend’s “absent = preserve / null = clear” contract, so webdev can silently diverge from native behavior for this new field.
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Build the save payload conditionally so `install_nag_dismissed_at` is truly absent in JS when preservation is intended, and update the mock `settings_save` merge path to ignore missing/`undefined` fields the same way the backend does.

### MEDIUM

- **[F003]** `src/crosshook-native/src/hooks/useOnboarding.ts:159` — `dismissUmuInstallNag()` has no failure handling, and the UI caller explicitly discards the returned promise. If the settings write fails, the button path devolves into an unhandled rejection with no user-visible error, so the dismiss action can appear broken.
  - **Status**: Fixed
  - **Category**: Completeness
  - **Suggested fix**: Catch the command failure in the hook, surface it through `checkError` (or dedicated UI error state), and only clear local guidance after the save succeeds.

- **[F004]** `src/crosshook-native/src/lib/mocks/handlers/onboarding.ts:60` — the new Flatpak `umu_install_guidance` branch is not exercisable in browser dev mode because the onboarding mock always returns `umu_install_guidance: null`. That leaves the new review-step UI and dismissal flow effectively unverified outside native mode.
  - **Status**: Fixed
  - **Category**: Completeness
  - **Suggested fix**: Add a mock fixture or toggle that returns representative `umu_install_guidance` data so browser-only verification can cover the new guidance row and dismissal flow.
