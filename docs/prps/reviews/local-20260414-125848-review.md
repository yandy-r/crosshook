## Summary

Broad a11y/semantics cleanup pass across 46 files — `role="button"` spans replaced with real `<button type="button">`, decorative SVGs get `aria-hidden="true"`, `<div role="list">` → `<ul>/<li>`, modal overlay roles restructured. Direction is correct, but several `role="img"` misapplications hide live-region/text content from AT, two modal Escape paths are non-functional, and `CollectionViewModal` now nests `<li>` inside `<li>` (invalid HTML). Recommend fixing all HIGH issues before commit.

### HIGH

- **[F001]** `src/crosshook-native/src/components/collections/CollectionViewModal.tsx:272` — `<li style={{ display: 'contents' }}>` wraps `LibraryCard`, which in this diff was changed to render as a native `<li>`. This produces `<ul><li><li>...</li></li></ul>`, which is invalid HTML. `display:contents` removes the box from layout but does not remove the element from the DOM or a11y tree; WebKitGTK behavior with invalid list nesting is not guaranteed. [correctness, quality]
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Remove the wrapper `<li>` entirely. Render `<LibraryCard>` directly inside the `<ul>`, mirroring how `LibraryGrid` does it (the card's own `<li>` becomes the direct grid child).

- **[F002]** `src/crosshook-native/src/components/layout/ControllerPrompts.tsx:13` — The outer `<div>` now has `role="img" aria-label="Controller shortcuts"` while its inner wrapper is `<div role="status" aria-live="polite">`. The ARIA spec treats a `role="img"` subtree as an opaque image; screen readers will suppress the inner live region, so dynamic controller-prompt updates are never announced. [security, quality]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Drop `role="img"` from the outer container and keep the existing `aria-label`. If the glyph row is purely decorative, mark it `aria-hidden="true"` instead, and move the label onto the inner `role="status"` element.

- **[F003]** `src/crosshook-native/src/components/HealthBadge.tsx:139` — The `onKeyDown` handler that manually calls `onClick?.()` on Enter and Space was ported from the old `span[role="button"]` to the new native `<button>`. Native buttons fire `click` on Enter at `keydown` and on Space at `keyup`; the manual handler causes `onClick` to run twice on Space activation in most browsers. [correctness]
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Remove the `onKeyDown` handler on the `<button>` variant entirely. Native button activation already covers Enter and Space.

- **[F004]** `src/crosshook-native/src/components/RunExecutablePanel.tsx:73` and `:236` — A `document.addEventListener('keydown', …)` registered in `useEffect` already closes the modal on Escape (and restores focus to the trigger). The diff additionally added an `onKeyDown` on the overlay `<div role="presentation">` that also calls `closeConfirmation()`. Both fire on Escape, causing `closeConfirmation` (and `triggerButtonRef.current?.focus()`) to run twice per keypress. [security, quality]
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Remove the overlay `onKeyDown`. The document-level listener is sufficient and already handles focus restoration on teardown.

- **[F005]** `src/crosshook-native/src/components/UpdateGamePanel.tsx:232` — Overlay `<div role="presentation">` has an `onKeyDown` Escape handler but no `tabIndex`, no document-level listener, and no focus-management on open. The inner dialog's `onKeyDown` calls `e.stopPropagation()`, so Escape events from inside the dialog never reach the overlay. Pressing Escape does not close this confirmation modal. [correctness, quality]
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Mirror the pattern used in `RunExecutablePanel`: register a `document.addEventListener('keydown', …)` inside a `useEffect`, focus a cancel button on open, and remove the overlay `onKeyDown` handler.

- **[F006]** `src/crosshook-native/src/components/UpdateGamePanel.tsx:242` — The new `role="dialog"` element is missing `aria-labelledby` and `aria-describedby`. The inner `<h4>` and `<p>` have no `id`. Screen readers will announce "dialog" with no name or description. [correctness, security, quality]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Add `id="update-game-confirm-title"` to the `<h4>` and `id="update-game-confirm-body"` to the `<p>`, then add `aria-labelledby="update-game-confirm-title" aria-describedby="update-game-confirm-body"` to the dialog `<div>`, matching the `RunExecutablePanel` pattern.

### MEDIUM

- **[F007]** `src/crosshook-native/src/components/ConfigHistoryPanel.tsx:87` — `<pre role="img" aria-label="Unified diff">` treats the entire diff block as an opaque image, suppressing all inner `<span>` elements (the colored diff lines) from AT. Screen reader users can no longer read any diff content. [correctness, security, quality]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Remove `role="img"` from the `<pre>`. Keep `aria-label` alone, or attach the label to a wrapping `<section>`/`<figure>` with `<figcaption>`.

- **[F008]** `src/crosshook-native/src/components/ExternalResultsSection.tsx:88` — The React `key` changed from a `source+sourceUrl+index` composite to `result.sourceUrl` alone. `ExternalTrainerResult.sourceUrl` is not guaranteed unique across sources (mirrored URLs are plausible), so duplicate keys are possible and would cause silent reconciliation bugs. [security]
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Use a composite key such as `` `${result.source}-${result.sourceUrl}` `` to restore source-scoped uniqueness without the index.

- **[F009]** `src/crosshook-native/src/components/LaunchOptimizationsPanel.tsx:453` — `<div role="img" aria-label="Launch optimization summary">` wraps `<span>` children whose text content (e.g. "3 enabled / 1 common / 2 advanced") is the information users need. `role="img"` suppresses the child text; AT announces only the static "Launch optimization summary" label with no counts. [correctness, security, quality]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Either remove `role="img"` and let the text content speak for itself, or build a dynamic `aria-label` that concatenates the counts and mark the visual chips `aria-hidden="true"`.

- **[F010]** `src/crosshook-native/src/components/LaunchPipeline.tsx:100` — Inner `<span role="img" aria-label="…">` is used as a Radix `Tooltip.Trigger asChild` child with no `tabIndex`, making the trigger non-focusable (tooltip unreachable via keyboard). `role="img"` additionally hides any text content from AT. [correctness]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Remove `role="img"` and either add `tabIndex={0}` to the span or convert the trigger to a `<button type="button">` with reset styles (matching the `InfoTooltip` pattern).

- **[F011]** `src/crosshook-native/src/components/LauncherExport.tsx:228` (also `:240` and `src/crosshook-native/src/components/SettingsPanel.tsx:666`) — `<label>` elements with no `htmlFor` were converted to `<span>`. The visual labels no longer have any programmatic association with the readonly inputs/display fields they describe, so AT users get no accessible name. [quality]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Add `id` to the inputs/display elements and restore `<label htmlFor="…">`, or add `aria-label`/`aria-labelledby` directly on the inputs.

- **[F012]** `src/crosshook-native/src/components/collections/CollectionAssignMenu.tsx` (and `HealthDashboardPage.tsx`, `ProtonDbOverwriteConfirmation.tsx`, `CompatibilityPage.tsx`, `CollectionViewModal.tsx`, `TrainerDiscoveryPanel.tsx`, `ExternalResultsSection.tsx`) — Repeated inline reset styles: `{ border: 0, margin: 0, padding: 0 }` on `<fieldset>` in four files and `{ listStyle: 'none', margin: 0, padding: 0 }` on `<ul>` in three files. This drifts away from the BEM `crosshook-*` convention and duplicates the list reset already added to `.crosshook-library-grid` in `library.css`. [quality]
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Add shared utility classes (e.g. `crosshook-fieldset-reset`, `crosshook-list-reset`) in `variables.css` or a new `utilities.css`, and replace the inline objects with class references.

- **[F013]** `src/crosshook-native/src/components/icons/SidebarIcons.tsx:81` — `InfoCircleIcon` now always renders with `aria-hidden="true"` and still contains a child `<title>Info</title>`. The `<title>` has no effect on an `aria-hidden` SVG and will mislead future readers who try to rely on it for labeling (the wrapping `<button aria-label="Info">` is now the sole accessible name). [quality]
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Remove `<title>Info</title>` from the SVG body. Consumers that need a name should wrap the icon in a labelled interactive element (as `InfoTooltip` already does).

- **[F014]** `src/crosshook-native/src/components/library/GameDetailsMetadataSection.tsx:66` — `<div role="img" aria-label="Genres">` wraps `<span>` chips whose text content (actual genre names like "Action", "RPG") is the entire point of the region. `role="img"` hides all child text from AT; users only hear "Genres". [quality]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Remove `role="img"`. The chips already convey text content. Use a visually-hidden label node or a `<dl>/<dt>/<dd>` pattern if a group label is desired.

- **[F015]** `src/crosshook-native/src/components/library/LibraryCard.tsx:91` — The element changed from `<div>` to `<li>`, carrying over an `onKeyDown` for Shift+F10 / ContextMenu. A plain `<li>` has no default focus behavior and the diff did not add `tabIndex`, so keyboard users cannot focus the card and the handler is unreachable. This is security-adjacent because the context menu launches profile operations. [security]
  - **Status**: Fixed
  - **Category**: Completeness
  - **Suggested fix**: Add `tabIndex={0}` to the `<li>` and restore a visible focus style (or attach the keyboard handler to the primary action button inside the card instead of the `<li>`).

- **[F016]** `src/crosshook-native/src/components/pages/CompatibilityPage.tsx:86` — The toggle group was changed from `<div role="group" aria-labelledby="…">` to `<fieldset>` + `<legend>`, but the group contains `<button>` elements, not form controls. `<legend>` semantics do not associate with `<button>` children, so no labeling benefit is gained over the previous ARIA approach while introducing non-idiomatic markup. [correctness]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Revert to `<div role="group" aria-label="Catalog source">` (or `aria-labelledby` pointing at a visible heading) for grouping non-input controls.

- **[F017]** `src/crosshook-native/src/components/pages/HealthDashboardPage.tsx:199` — `SummaryCard` count `<div role="img" aria-label={disabled ? undefined : ...}>`. When `disabled` is true the role remains but `aria-label` is undefined, so AT announces an unnamed "image" where the "—" placeholder text used to be readable. [correctness, quality]
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Drop `role="img"` when `disabled` is true (or always), and let the placeholder text be announced directly. If a composite label is desired, produce a non-undefined `aria-label` unconditionally.

- **[F021]** `src/crosshook-native/src/components/LaunchPipeline.tsx:94` — The `<li aria-label="…">` and its new inner `<span role="img" aria-label="…">` carry identical labels. AT will announce the label twice on focus. [quality]
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Remove `role="img"` and `aria-label` from the inner span; the parent `<li aria-label>` already names the step.

- **[F022]** `src/crosshook-native/src/components/OfflineReadinessPanel.tsx:61` (also `src/crosshook-native/src/components/ReadinessChecklist.tsx:95`, `src/crosshook-native/src/components/wizard/WizardReviewSummary.tsx:76`) — `key` was simplified away from `field+message+index` to `field+message` (or `field`) alone.
