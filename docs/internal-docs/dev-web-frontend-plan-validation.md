## NEEDS_WORK / INCOMPLETE Tasks — Detailed Findings

### Task 2.6: Update flow handlers + events

**Issue**: The only behavioral specification is "Pattern matches install flow." The `cancel_update` command requires event-suppression logic (clearing pending `setTimeout` emissions) per `analysis-tasks.md` ("a simple 'cancelled' flag in the module is sufficient"), but this requirement is absent from the parallel-plan task body. An implementer will produce a naive `cancel_update` that mutates a status field but does not suppress the in-flight mock streaming events, causing the UI to show update-complete after the user cancelled.

**Suggestion**: Add one sentence: "For `cancel_update`: set a module-level `let cancelled = false` flag at the start of `update_game`, check it before each `setTimeout` emission, and set it `true` from `cancel_update` so pending streaming events abort cleanly." Also add the exact event sequence: `update-log` (streaming), `update-complete`.

### Task 2.9: ProtonDB lookup handlers

**Issue 1**: The overlap with Task 2.3 (`health.ts` includes `protondb_lookup` through `protondb_dismiss_suggestion`) is unacknowledged in this task body. Two parallel implementers could register the same command keys in different handler files, producing duplicate registrations in `mocks/index.ts` (last-write wins; earlier registration silently dropped).

**Issue 2**: `protondb_accept_suggestion` and `protondb_dismiss_suggestion` presumably need to update some store field (e.g., mark a suggestion dismissed) or at least acknowledge the call. No guidance is given on whether to mutate state, emit an event, or simply `return null`.

**Suggestion**: Add "Coordinate with Task 2.3: if `protondb_*` commands were added to `health.ts`, move them to `protondb.ts` in this PR and update `health.ts`." Add a sentence: "`protondb_accept_suggestion` and `protondb_dismiss_suggestion` may return `null` and are non-event-emitting for Phase 2; add a `dismissedSuggestions: Set<string>` field to the store if needed for `protondb_get_suggestions` to filter them."

### Task 2.11: Launcher export handlers

**Issue 1**: `validate_launcher_export` return type is unspecified. Analysis-tasks.md says it "returns success with a preview payload" — the preview payload type needs to come from `types/launcher.ts` (in the READ list), but the instruction gives no guidance on what to return.

**Issue 2**: `delete_launcher*` uses a wildcard. The actual command names need to be discovered from `useLauncherManagement.ts` or `types/launcher.ts`. For an implementer who needs to register exact `map.set('delete_launcher_entry', ...)` keys, the wildcard is ambiguous.

**Issue 3**: No `export_launchers` return type is specified. "Log a warn but return success" tells the implementer to succeed, but "success" for this command could be a boolean, a string, or a typed payload.

**Suggestion**: Add: "Discover exact command names from `src/crosshook-native/src/hooks/useLauncherManagement.ts` — look for all `callCommand('...')` call sites in that file. Return types are in `types/launcher.ts`. `export_launchers` can return `{ exported: number, path: '/mock/launchers' }` or whatever the `LauncherExportResult` type requires."

**Issue 1 (minor rendering artifact)**: Transform 4 contains a Markdown bold-underline collision: `` className={`crosshook-app crosshook-focus-scope${**WEB_DEV_MODE** ? ...}` `` — the double asterisks are Markdown for `**bold**`, rendering as bold in any Markdown viewer and stripping the underscores, producing `WEB_DEV_MODE` instead of `__WEB_DEV_MODE__`. ## Cross-Cutting Observations

- **Phase 2 terseness is intentional and generally acceptable.** The uniform `register<Domain>(map)` pattern is established in Phase 1, and all Phase 2 tasks correctly include the relevant type-file source in the READ list. GOOD ratings across most Phase 2 tasks reflect appropriate compression.

- **The "READ THESE BEFORE TASK" lists are consistently high-quality.** Every task includes the exact files needed. There are no cases where a critical type file or context doc was omitted from a READ list.

- **Phase 2 overlap between 2.3 and 2.9** (ProtonDB commands listed in both health.ts and protondb.ts tasks) is the single highest-risk coordination gap for parallel execution. If both tasks are executed simultaneously by different contributors, `lib/mocks/index.ts` will get duplicate `registerProtonDb(map)` / `registerHealth(map)` wiring for the same command keys.

- **The `analysis-tasks.md` document provides significantly more Phase 2 detail** (specific command names, event names, store field guidance, per-domain notes) than the parallel-plan task bodies. Phase 2 implementers should be explicitly directed to read `analysis-tasks.md` alongside the parallel-plan — this cross-reference exists in the "Critically Relevant Files" preamble of the parallel-plan but is not repeated in individual Phase 2 task headers.

- **Phase 3 tasks 3.1 and 3.2 have a latent sequencing issue.** The `wrapHandler()` middleware concept (needed for `?loading` fixture's never-resolving promise and `?delay` wrapping) lives in 3.2 but is logically required by 3.1's `loading` fixture implementation. A `loading` fixture that manually returns `new Promise(() => {})` in every affected handler (rather than using a centralized wrapper) is functionally equivalent but structurally divergent from what 3.2 will later build. Either merge 3.1 and 3.2 into one task, or document in 3.1 that `loading` should use a simple per-handler `if (fixture === 'loading') return new Promise(() => {})` until 3.2's `wrapHandler` middleware is available.

- **`isTauri()` implementation has a minor discrepancy between documents.** `feature-spec.md` (API Design section) defines `isTauri()` as checking `globalThis.isTauri` (the Tauri v2 official probe), while `analysis-tasks.md` dependency graph defines it as `'__TAURI_INTERNALS__' in window`. The parallel-plan task 1.4 correctly uses `'__TAURI_INTERNALS__' in window` consistent with the Tauri v2 bridge behavior, and cross-references `research-technical.md` for confirmation. The feature-spec.md version (`globalThis.isTauri`) is a secondary Tauri v2 property that may or may not be set depending on the Tauri version. Task 1.4 is correctly specified; the feature-spec.md discrepancy is a documentation inconsistency that should be noted but does not affect implementation.

- **No task references `shared.md` or `analysis-context.md`** in its READ list, even though these were listed as "Critically Relevant Files" in the plan preamble. This is acceptable — both are synthesis documents that point to the primary sources, and the primary sources (research-technical.md, research-business.md, feature-spec.md) are consistently referenced.

- **All 18 Phase 1 tasks correctly terminate with a success criterion or verification step** (either explicit verification commands or implied by the "files to create/modify" specificity). This is the strongest structural property of the plan.
