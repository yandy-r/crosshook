**Task 1.5: Modify profile_delete to cascade launcher deletion** -- NEEDS MINOR IMPROVEMENT

- Purpose is clear, scope is correct (1 file)
- The cascade being best-effort is well-emphasized
- Issue 1: The instruction says to call `resolve_display_name(&profile.steam.launcher.display_name, &profile.steam.app_id, &profile.trainer.path)` but the actual function signature is `resolve_display_name(preferred_name, steam_app_id, trainer_path)`. The field mapping is correct (`steam.launcher.display_name` -> `preferred_name`, `steam.app_id` -> `steam_app_id`, `trainer.path` -> `trainer_path`), but this does NOT match how the existing `export_launchers()` calls it -- that function passes `request.launcher_name`, not `profile.steam.launcher.display_name`. A developer needs to understand that the profile field `steam.launcher.display_name` corresponds to what the export request calls `launcher_name`. This mapping is subtly correct but should be explicitly documented.
- Issue 2: The instruction says to check `profile.launch.method` is NOT `"native"`, but the Rust field is `profile.launch.method` which is a `String`. The task should note the comparison format (`!= "native"` as a string comparison in Rust, not a TypeScript enum check).
- Issue 3: `resolve_target_home_path("", "")` is called with empty strings, which will fall through to the `$HOME` env var fallback -- this is correct behavior per the function at line 465-489, but this implicit behavior should be more explicitly noted for the implementer.
- Issue 4: The task says "Import the necessary functions from `crosshook_core::export`" but `resolve_display_name` is currently private and will become `pub(crate)` after Task 1.1 -- that means it can only be imported within the `crosshook_core` crate. Since `profile.rs` lives in `src-tauri` (a different crate), the import `crosshook_core::export::launcher::resolve_display_name` will FAIL because `pub(crate)` is not visible across crate boundaries. The implementer needs to either: (a) re-export `resolve_display_name` as fully `pub` from `crosshook_core::export`, (b) add a `pub` wrapper function in `launcher_store` that derives the slug from profile fields, or (c) inline the derivation logic. This is a real blocker that the plan does not address.

**Task 1.6: Add launcher status indicator to LauncherExport.tsx** -- NEEDS MINOR IMPROVEMENT

**Task 2.4: Add rename detection to useProfile.ts** -- NEEDS MINOR IMPROVEMENT

- Purpose is clear, scope is correct (1 file)
- The rename detection condition `profileName.trim() !== selectedProfile && selectedProfile !== '' && profiles.includes(selectedProfile)` is correct -- it detects when the user has changed the name field while an existing profile is loaded
- Issue 1: The task says "line ~295" for `saveProfile()` -- the actual line is 295 (exact match, confirmed)
- Issue 2: After calling `profile_rename`, the task calls `refreshProfiles()` then `loadProfile(profileName.trim())` and returns. But the existing `saveProfile` flow also calls `normalizeProfileForSave` and `invoke('profile_save', ...)`. If rename happens, the new profile TOML retains the old content -- but the user may have also edited profile fields. The rename-then-return flow skips saving the edited content. The task should specify whether a rename-and-save should be a two-step operation (rename first, then save updated content to the new name) or whether content edits during rename are discarded.

The Tauri command receives only `old_name` and `new_name` strings -- it loads the profile from disk to get display name data. If the user edited fields (like `display_name`) but hasn't saved yet, the rename cascade uses the old on-disk data, not the in-memory edits. This is fine as designed but should be documented.

## Priority Improvements

1. **[BLOCKER] Task 1.5 -- pub(crate) visibility cross-crate access**: `resolve_display_name` elevated to `pub(crate)` in `crosshook-core` cannot be called from `src-tauri` commands (different crate). The cascade in `profile_delete` (which lives in `src-tauri`) needs to derive slugs. Solution: add a `pub` function in `launcher_store` like `pub fn derive_launcher_slug_from_profile(profile: &GameProfile) -> String` that calls the `pub(crate)` functions internally, or make `resolve_display_name` fully `pub`.

2. **[BLOCKER] Tasks 1.4/1.6 -- Slug derivation mismatch**: Task 1.4 defines `check_launcher_exists` accepting a `launcher_slug: String`, but Task 1.6 acknowledges the frontend cannot derive slugs. Either change `check_launcher_exists` to accept profile-level fields (display_name, app_id, trainer_path) and derive the slug internally, or add a `derive_launcher_slug` Tauri command. Both tasks need to agree on the approach.

3. **[IMPORTANT] Task 2.4 -- Rename + save interaction**: The rename-then-return flow skips saving any content edits the user made alongside the name change. Specify explicitly: should rename discard content edits (requiring a separate save), or should the task save content to the new name after renaming?

4. **[MINOR] Task 2.6 -- Stale/rename detection mechanism**: Clarify how the frontend detects slug mismatch without a local slug derivation function. Specify the concrete comparison mechanism.

5. **[MINOR] Task 3.1 -- Signature change**: Note that `build_desktop_entry_content` needs `profile_name` and `slug` added to its parameters, requiring all callers to be updated.

6. **[MINOR] Task 3.3 -- Design decision**: Commit to either modifying `check_launcher_exists` or adding a separate `check_launcher_stale` function, rather than leaving it as an either/or.

7. **[MINOR] Task 2.1 -- Overwrite guard**: Consider whether `ProfileStore::rename` should fail if `new_path` already exists (preventing accidental overwrites of existing profiles).

## Overall Assessment

This is a high-quality implementation plan. The level of detail is exceptional -- line numbers, exact function signatures, color hex values, serde derive patterns, and file paths are all verified as accurate against the actual codebase. The phasing is logical, dependencies are correctly identified, and the advice section catches most non-obvious gotchas (slug collision, hardcoded paths, native method skip, lib.rs coordination).

The two blockers (items 1 and 2 above) both stem from the same root issue: the plan elevates functions to `pub(crate)` for intra-crate use in `launcher_store`, but then has Tauri commands (in a separate crate) that also need slug derivation. The fix is straightforward -- expose a `pub` slug-derivation facade in `launcher_store` that wraps the `pub(crate)` chain -- and the plan nearly arrives at this conclusion in its own note on Task 1.6. These should be resolved before implementation begins, but the fixes are additive, not disruptive.

All 19 tasks have clear purpose, specific file paths, and actionable instructions. No task exceeds 3 files in scope. The "READ THESE BEFORE TASK" sections with specific document references and section names are a strong pattern that reduces context-gathering time. The plan is ready for implementation after addressing the two blockers.
